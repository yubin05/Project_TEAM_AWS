# 로그 담당

## 가상 고객 요구사항

### 요구사항 1. 트래픽 급증 시 서비스 성능 저하

**시나리오:**
여행 예약 플랫폼 '여기어때'는 여름 성수기, 연휴 시즌에 신규 프로모션 이벤트를 진행합니다. 이벤트 시작과 동시에 평소보다 5배 이상 트래픽이 몰리면서 서버 과부하로 예약 페이지가 느려지거나 다운되는 장애가 발생했습니다. 고객들이 예약을 못 하고 이탈하는 사태가 반복되면서 매출 손실이 발생했습니다.

**고객 요구사항:**
- 트래픽 급증 시 서버가 자동으로 확장되어 성능 저하 없이 서비스를 유지해 주세요
- 전 세계 사용자에게 빠른 콘텐츠 제공이 필요합니다

**해결 방안:**
- ASG, ELB를 통해 자동으로 트래픽을 분산하여 성능 저하 방지
- CloudFront를 통해 웹 페이지 및 각종 데이터를 전 세계적으로 빠르게 제공
- CloudWatch 알람으로 CPU 70% 초과 시 즉시 알림 및 자동 스케일 아웃

---

### 요구사항 2. 데이터베이스 부하 및 장애

**시나리오:**
예약이 집중되는 시간대에 DB에 갑작스러운 높은 부하가 발생하거나 장애가 생길 경우, 예약/결제 데이터 전체가 영향을 받아 서비스 전체가 중단될 우려가 있습니다. 현재 RDS MySQL 단일 인스턴스로는 장애 발생 시 수동 복구가 필요하고 복구 시간이 길어 고객 불만이 증가했습니다.

**고객 요구사항:**
- DB 장애 시 서비스가 자동으로 복구되어야 합니다
- 조회 트래픽이 많아 DB 부하가 심하니 분산 처리가 필요합니다

**해결 방안:**
- Aurora(고가용성 DB)를 사용하여 장애 시 자동으로 복구
- Aurora Global DB로 서울-도쿄 멀티 리전 구성, 리전 장애 시 1초 미만 자동 Failover
- 읽기 전용 복제본 추가로 조회 트래픽 분산

---

### 요구사항 3. 데이터 손실 및 보안 위협

**시나리오:**
여행 예약 플랫폼에서 고객의 예약 정보, 결제 내역, 개인정보가 저장되어 있습니다. 랜섬웨어 공격이나 운영자 실수로 데이터가 삭제되거나 변조될 경우 고객 정보 및 주문 내역 손실로 이어져 큰 피해가 발생할 수 있습니다. 또한 해킹 시도나 비정상적인 API 호출에 대한 탐지가 현재 불가능한 상태입니다.

**고객 요구사항:**
- 데이터 손실에 대비한 자동 백업 및 복구 체계가 필요합니다
- 보안 위협을 실시간으로 탐지하고 알람을 받고 싶습니다
- 로그는 비용 효율적으로 장기 보관이 필요합니다

**해결 방안:**
- S3 버킷을 통한 데이터 백업을 정기적으로 수행하여 안전하게 저장
- AWS Backup으로 백업 및 복구 계획을 자동화하고 재해 복구 전략 수립
- GuardDuty를 통해 비정상 API 호출, 계정 탈취 시도 자동 탐지 후 CloudWatch 알람 연동
- S3 Lifecycle Policy로 30일 이후 로그를 Glacier로 자동 이동하여 비용 절감 + 1년 이상 보관

---

### 요구사항 4. 해외 고객 대상 서비스 제공 시 성능 저하

**시나리오:**
여행 예약 플랫폼이 일본, 동남아 등 해외 고객을 대상으로 서비스를 확장하면서 특정 지역에서 페이지 로딩이 느리고 예약 완료까지 시간이 오래 걸리는 문제가 발생했습니다. 특히 호텔 이미지 로딩이 느려 고객 이탈률이 높아졌습니다.

**고객 요구사항:**
- 해외 사용자에게도 빠른 서비스 응답 속도가 필요합니다
- 외국어 사용자를 위한 다국어 지원이 필요합니다

**해결 방안:**
- CloudFront를 사용해 글로벌 사용자에게 빠르게 콘텐츠 제공
- Route53을 이용해 최적의 DNS 라우팅 설정
- Global Accelerator를 통해 AWS 백본 네트워크로 트래픽을 라우팅하여 응답 속도 개선
- Azure Translator를 통해 호텔 정보를 다국어로 자동 번역하여 제공

---

### 요구사항 5. 예약 프로세스 실패 시 롤백 불가

**시나리오:**
고객이 예약을 진행할 때 예약 확인 → 객실 잠금 → 결제 → 이메일 발송 순서로 처리됩니다. 중간 단계에서 오류가 발생할 경우 이미 처리된 단계가 롤백되지 않아 객실은 잠겼는데 결제는 안 되거나, 결제는 됐는데 이메일이 발송 안 되는 문제가 발생했습니다. 고객 민원이 급증하고 있습니다.

**고객 요구사항:**
- 예약 프로세스 중간에 오류가 생겨도 자동으로 롤백되어야 합니다
- 예약 처리 전 과정을 모니터링하고 싶습니다

**해결 방안:**
- Step Functions으로 예약 워크플로를 단계별로 관리
- 각 단계 실패 시 자동 롤백 구성
- SQS로 메시지 큐를 구성해 비동기 처리로 안정성 강화
- CloudWatch로 예약 프로세스 전 과정 모니터링 및 실패 알람 연동

# 배포 자동화 담당

## 시나리오 구성

여행 플랫폼 기업 'Travel-On' 은
1) 기존 단일 서버(EC2 1대) 기반의 레거시 구조는 성수기 트래픽 폭증을 버티지 못해 서비스 전체 다운이 반복되고, 개발자가 직접 SSH로 접속해 수동 배포하는 방식으로 인해 배포 오류 및 롤백 불가 사고가 누적되고 있습니다.
2) Travel-On은 단순히 서버를 이전하는 수준이 아닌, 고가용성·보안성·가시성이 확보된 전면적인 클라우드 네이티브 전환을 의뢰했습니다.

## 가상 고객 요구 사항

1) 고가용성
문제: 단일 서버 장애 시 전체 서비스 중단, 성수기 트래픽 폭증 대응 불가
해결방안: ECS Fargate + ALB로 컨테이너 이중화, 특정 서비스 장애 시에도 나머지 서비스는 정상 운영 유지

2) 보안
문제: DB 접속 정보·API 키가 코드에 평문 노출, 외부 공격 차단 체계 없음
해결방안: Secrets Manager로 민감 정보 코드에서 완전 분리, WAF + Cognito로 인증·공격 차단

3) 비용 최적화
문제: 수동 배포로 인한 인건비 낭비, 변경 없는 영역까지 재빌드되어 불필요한 빌드 비용 발생
해결방안: CodePipeline V2 경로 필터로 변경된 영역만 선택 빌드, Amplify로 프론트 배포 분리하여 불필요한 빌드 차단

## 시나리오 구성

1. 과거 경쟁 업체에서 관리자 계정 탈취로 인해 고객들에게 가짜 결제 링크가 발송되는 사고가 발생. 이에 따라 관리자 계정 보호와 로그 추적 체계를 강화하려고 함
2. 과거 스타우드 호텔 침해 사건처럼 예약 시스템이 장기간 침해되면 고객의 민감정보가 대량 유출될 수 있어 개인정보 보호, 웹 공격 방어, 계정 보호, 비밀정보 관리, 이상행위 탐지를 필수 요구사항으로 제시함

## 가상 고객 요구 사항

1. 예약 DB 직접 접근 차단
2. 웹 공격 차단
3. 권한 최소화
4. 침해 장기 미탐지 방지
5. 민감정보 보호

**요구 사항 해결법:**

예약 DB 직접 접근 차단
- DB는 Private Subnet에 배치하여 외부 접근 불가
- 사용자는 Amplify을 통해 예약 기능을 사용할 수 있음
- Cognito로 안전한 로그인 보장

웹 공격 차단
- WAF 로 SQL Injection, XSS, Known Bad Inputs 차단
- API Gateway 또는 ALB 에 WAF 적용 (우리는 ALB에 적용함)

권한 최소화
- IAM은 담당자별 최소 권한 부여
- 인프라, 배포, 보안, 로그 담당 권한 분리
- 관리자 권한 계정은 잠금

침해 장기 미탐지 방지
- CloudTrail로 API 호출 기록 저장
- GuardDuty로 이상행동 탐지
- Inspector로 서버 취약점 점검
- Macie로 개인정보 유출 탐지

민감정보 보호
- Secrets Manager로 DB 비밀번호 보호
- AWS Config로 보안 정책 준수

## 시나리오 구성

Sponge Trip은 한국 내 여행 및 숙박 예약 서비스로 시작했으나, 바이럴 이후 일본·동남아시아 등 해외 사용자 유입이 급증하며 Sponge Company는 글로벌 서비스 확장을 검토 중이다.

현재 고객사는 온프레미스 데이터 센터에 아래와 같이 서버를 운영 중이다.

| 서버 | 사양 | 역할 |
|------|------|------|
| 웹 서버 1대 | Nginx | 프론트엔드 정적 파일 서빙 |
| 백엔드 API 서버 4대 | Docker 컨테이너 | 숙소 검색·예약·리뷰·인증 API |
| DB 서버 1대 | MySQL | 예약·사용자·숙소 데이터 저장 |

### 인기 폭발 배경

2026년 초, 구독자 150만 명의 국내 유명 여행 유튜버 채널에서 Sponge Trip을 소개하는 영상이 업로드되었다. 해당 영상은 24시간 만에 조회수 800만을 돌파하며 SNS 전방위로 바이럴됐고, 신규 가입자와 예약 요청이 폭발적으로 증가했다.

| 지표 | 바이럴 이전 | 바이럴 직후 |
|------|------------|------------|
| 일 활성 사용자(DAU) | 약 5,000명 | 약 200,000명 (40배 ↑) |
| 동시 접속자 | 약 500명 | 약 20,000명 |
| 초당 예약 요청 | 약 10건 | 약 300건 |

기존 온프레미스 서버는 급증한 트래픽을 감당하지 못해 **서비스 전체가 약 4시간 다운**되었고, 이로 인해 예약 기회 손실 및 브랜드 신뢰도 타격이 발생했다.

이에 Sponge Company는 **클라우드 기반의 고가용성 인프라** 구축을 결정했다.

## 가상 고객 요구사항

### 고가용성 및 완전관리형 전환

고객사는 성수기나 이벤트 기간 트래픽 급증 시 프론트엔드·백엔드·데이터베이스 전 레이어가 다운 없이 자동 대응되기를 원한다.
또한 서버 패치, 스케일링, 장애 복구 등을 직접 관리하는 오버헤드 없이 **완전관리형(Fully Managed)** 서비스로 운영되기를 요구한다.

| 레이어 | 기존 (온프레미스) | 전환 후 (AWS) | 이점 |
|--------|-----------------|--------------|------|
| 프론트엔드 | Nginx (수동 관리) | **Amplify** | GitHub 연동 자동 배포, 내장 CDN, 서버 관리 불필요 |
| 백엔드 | Docker (수동 운영) | **ECS Fargate** | 컨테이너 오케스트레이션 자동화, 트래픽에 따른 Auto Scaling |
| 데이터베이스 | MySQL (단일 서버) | **Aurora Serverless v2** | 트래픽 급증 시 자동 확장, 자동 Failover (30초 이내) |

- 프론트엔드: Amplify CDN이 전 세계 엣지에서 정적 파일 서빙 → 서버 부하 없이 대용량 트래픽 처리
- 백엔드: ECS Fargate Auto Scaling으로 트래픽 증가 시 컨테이너 자동 증설, 감소 시 자동 축소
- 데이터베이스: Aurora Serverless v2 — 0.5 ACU ~ 4 ACU 자동 조절, Writer 장애 시 Reader → Writer 자동 승격 (30초 이내)

### 글로벌 서비스 확장

국내 바이럴 이후 일본·동남아시아 사용자 유입이 늘어나며 고객사는 해외 시장 진출을 원한다.

- 일본·동남아 등 해외 사용자에게도 국내와 동일한 응답 속도를 제공해야 한다.
- 특정 국가의 클라우드 장애 시에도 다른 리전에서 서비스가 유지되어야 한다.
- 글로벌 사용자를 고려해 정적 파일은 CDN 엣지에서 빠르게 서빙되어야 한다.

> **현재 구현 범위**: 서울 리전(ap-northeast-2) 단일 구성
> **향후 고려사항** *(일정 여유 시)*: 도쿄 리전(ap-northeast-1)에 비용 절감 목적으로 최소 구성(단일 서브넷, 단일 RDS)으로 우선 확장 → 트래픽 증가 시 HA 업그레이드

### 재해복구 (AWS 장애 대비 멀티클라우드)

고객사는 특정 클라우드 리전 장애 시에도 서비스가 중단되지 않기를 원한다.

- 단일 클라우드 의존을 탈피하여 AWS 장애 시 Azure로 자동 전환되어야 한다.
- 정상 시에도 AWS와 Azure가 동시에 트래픽을 처리하는 Active-Active 구성이어야 한다.
- AWS 장애 발생 시 60초 이내 자동 전환(RTO), 데이터 손실은 5분 이내(RPO)로 최소화해야 한다.
- 이미지·정적 파일은 S3와 Azure Blob Storage 간 양방향 동기화로 어느 쪽에서든 제공되어야 한다.

```
글로벌 사용자
      │
      ▼
Route 53 (AWS) ──────── Azure Traffic Manager
      │                        │
      ├── Amplify (CDN)        ├── Azure Static Web Apps (CDN)
      │   프론트엔드           │   프론트엔드 이중화
      │                        │
      ▼                        ▼
API Gateway (AWS)     Azure API Management
  ↓ VPC Link              ↓
ALB → ECS (AWS)       Azure Container Apps
  ↓                        ↓
Aurora MySQL (AWS)  ←──→  Azure Database for MySQL
                    양방향 동기화
S3 (AWS)            ←──→  Azure Blob Storage
                    Storage Replication
```

| 구분 | 내용 |
|------|------|
| 정상 시 | Route 53이 AWS(70%) + Azure(30%) 가중치 라우팅 |
| 장애 시 | Route 53 헬스체크 실패 → Azure 100% 자동 전환 |
| RTO | 60초 이내 (DNS TTL 기반) |
| RPO | 5분 이내 (DB 동기화 주기) |

## 실제 사례

#### 야놀자 개인정보 유출 (2019, 2021)
- 2019년: 야놀자펜션앱 DB 해킹으로 고객 개인정보 **약 7만 건** 유출 (이메일, 전화번호)
- 2021년: 클라우드 관리 소홀(접근 권한 미설정)로 **5만 2천 건** 추가 유출
- 참고: [전자신문](https://www.etnews.com/20190328000230) / [보안뉴스](http://www.boannews.com/media/view.asp?idx=78254)

#### Booking.com 피싱 공격 (2023~2024)
- 2023년부터 호텔 파트너를 대상으로 한 악성코드 공격 → 고객 예약 정보 탈취 후 피싱 메시지 발송
- 2024년: Booking.com 발표 — 피싱 공격 **900% 증가** (AI 도구 활용)
- 피해: 영국에서만 피해액 **£370,000(약 6억 원)**
- 참고: [Krebs on Security](https://krebsonsecurity.com/2024/11/booking-com-phishers-may-leave-you-with-reservations/) / [Malwarebytes](https://www.malwarebytes.com/blog/data-breaches/2026/04/booking-com-breach-gives-scammers-what-they-need-to-target-guests)

#### Agoda 8,200만 건 유출 의혹 (2025)
- 해커 포럼에 고객 정보 **8,200만 건** 판매 게시 (이름, 이메일, 전화번호, 신분증 번호)
- Agoda 측은 부인했으나 샘플 데이터의 실존 여부 논란 지속
- 참고: [Cybernews](https://cybernews.com/security/agoda-data-breach-82m-hacker-forum/)

#### AWS 서울 리전 장애
